 1. Define a progress event contract.
     Create a shared type for something like:

  - requestId or correlationId
  - message
  - percent

  2. Add renderer subscription support in preload.ts.
     Right now you only expose invoke.
     You also need something like:

  - on(channel, listener)
  - off(channel, listener)
    or a dedicated subscribeCreateProjectProgress(...).

  3. Send progress from the main process to the renderer.
     This is the missing core step.
     You need access to the invoking renderer’s webContents, usually via the event from IPC.

  At the moment, safeHandle.ts gives the handler the event, but createProject.ts only receives ctx, not the event
  or a sender function.

  So you need one of these designs:

  - extend RequestContext to include a sendProgress(...) function
  - or pass event.sender down into the service
  - or emit progress in the IPC handler layer instead of inside the service

  I would prefer extending ctx with a narrow function like sendProgress(payload) rather than passing Electron
  objects deep into services.

  4. In onProgress, send IPC events to the renderer.
     Instead of only logging, do both:

  - log progress
  - send progress event to renderer

  5. Correlate progress with the correct request.
     This matters if multiple project creations could happen over time.
     You already have ctx.correlationId, which is a good candidate.
     Include it in the pushed progress payload so the frontend knows which active creation flow the update belongs
     to.
  6. In the frontend, subscribe before starting creation.
     Your modal or its hook should:

  - start listening for progress events
  - trigger createProject
  - update overlay state when matching progress events arrive
  - stop listening on success/error/unmount

  7. Render the live progress in ProcessingOverlay.
     Right now the overlay is hardcoded:

  - "Processing"
  - 20

  Instead, bind it to actual live state from the subscription.

  One important constraint:
  ipcRenderer.invoke(...) returns only once, at the end. So intermediate progress cannot come back through the
  createProject return value. You need a second IPC mechanism for progress events.
