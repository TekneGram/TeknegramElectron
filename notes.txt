For your current feature, I would probably keep:

  - create as a mutation hook
  - cancel as a direct service call inside useCreateProjectFlow

  Why?
  Because cancel is subordinate to the main create flow. It is not really an independent screen action; it is a
  control action within the running workflow.

  So the important architectural point is:

  - yes, services are where adapter/port interaction should happen
  - and that is already true for both create and cancel in your code
  - the only difference is whether React Query wraps the service call

  So this is not an oversight. It is just an uneven orchestration shape.

  If you want maximum consistency, add a cancel mutation hook.
  If you want pragmatic simplicity, your current service-based cancel path is fine.
